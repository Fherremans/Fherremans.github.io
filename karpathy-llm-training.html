Course Map

What is actually done when training an LLM

The lectures reduce modern LLMs to a concrete pipeline: collect text, convert it into tokens, train a decoder-only Transformer to predict the next token, scale the optimization, evaluate it, then post-train it into an assistant.

Data

Build a large document corpus, filter it, deduplicate it, and split it into train and validation shards.

Tokens

Encode text into integer IDs using a tokenizer, commonly byte-level BPE or a related subword method.

Model

Use token embeddings, positional information, causal self-attention, MLPs, residuals, and layer norms.

Optimize

Minimize next-token cross entropy with AdamW, schedules, mixed precision, and distributed training.

Align

Continue training on conversation, preference, or verifiable task data so the model behaves like an assistant.

Core Idea

The LLM pipeline in one pass

A base LLM is not hand-written. It is a large differentiable program whose parameters are tuned until its next-token probabilities match a massive text distribution. Assistant behavior is layered on later.

Training data flow

Documents HTML, books, code, papers, forums, filtered web text.
->
Token stream Integer IDs such as 15339, 1917, 220, with document separators.
->
Batches Windows shaped B x T, each position predicting the next token.
loss = mean_t -log p_theta(token[t+1] | token[0:t]) logits shape = batch_size x context_length x vocab_size

Decoder-only Transformer

Token IDs + position IDsLookup tables produce dense vectors.
Residual streamThe model refines one vector per token through many blocks.
LayerNorm -> causal self-attention -> residual addTokens read only earlier tokens through a triangular mask.
LayerNorm -> MLP/GELU -> residual addEach token computes locally after communication.
Final LayerNorm + LM headProject each token vector to vocabulary logits.
Course Notes

Technical summary by module

These modules are ordered as an implementation path: start with text and end with a trained base model, then layer on the extra data and objectives used for chat.

Module 01

Text becomes tokens before it becomes tensors

The model never sees characters, words, or web pages directly. It sees a sequence of token IDs. A tokenizer chooses the atoms of the sequence and therefore controls the effective context length, vocabulary size, and many failure modes.

Byte-level BPE

Start from UTF-8 bytes, so every text string can be represented. Repeatedly find the most frequent adjacent pair and mint a new token for that pair. The merge table is the tokenizer training result.

Vocabulary tradeoff

A larger vocabulary shortens sequences but makes the embedding table and final softmax wider. GPT-2 used 50,257 tokens; later tokenizers often use larger vocabularies and handle code whitespace more efficiently. 

# BPE training sketch
tokens = utf8_bytes(text)             # IDs 0..255
merges = []
while vocab_size < target_vocab_size:
    pair = most_frequent_adjacent_pair(tokens)
    new_id = next_id()
    tokens = replace_pair(tokens, pair, new_id)
    merges.append((pair, new_id))

Why this matters

Tokenization explains surprising behavior: spelling and reversing strings are awkward because the model sees chunks, not letters; arithmetic is awkward because digits are grouped inconsistently; non-English text may consume more tokens; code improves when indentation and syntax tokenize densely; special tokens and partial tokens can become API edge cases.

Module 02

The dataset is a product, not a folder of text

Karpathy frames pretraining as knowledge acquisition from text. The hard part before optimization is building a high-quality sequence: crawl, extract, filter, deduplicate, remove undesirable content, remove obvious PII, classify languages, tokenize, shard, and hold out validation data.

  • Common Crawl-like web data is noisy HTML; text extraction is a major preprocessing step.
  • URL and domain filters remove spam, malware, adult content, and low-value pages.
  • Language filters decide whether the model is English-heavy or multilingual.
  • Deduplication prevents repeated pages from wasting capacity and contaminating evaluation.
  • Documents are often concatenated with an end-of-text separator token.
  • Training shards must be shuffled carefully; repeated document order can show up in loss curves.

Technical consequence

If two models have the same architecture and token budget, the one trained on cleaner, denser, better-deduplicated data can win. This is why a modern reproduction can look surprisingly strong against an older GPT-2 checkpoint even with fewer tokens.

Module 03

Next-token prediction supplies the labels for free

A raw token stream contains its own supervised dataset. For a window of length T+1, the first T tokens are inputs and the next T shifted tokens are targets. Every position in the window is a training example. 

# Batch construction sketch
ix = random_offsets(batch_size)
x = stack(token_stream[i : i + T] for i in ix)
y = stack(token_stream[i + 1 : i + T + 1] for i in ix)

logits = model(x)                     # B x T x V
loss = cross_entropy(logits, y)       # average over B*T positions

Teacher forcing

The model is trained on real previous tokens, not its own generated tokens.

Validation loss

A held-out token stream estimates whether the model learned general structure or memorized.

Generation

At inference, sample one token, append it, crop to the context window, and repeat.

Module 04

The GPT block alternates communication and computation

In Karpathy's implementation, attention is where tokens exchange information; the MLP is where each token transforms its own representation. Residual connections keep a clean path for gradients, and pre-layer normalization makes deep training more stable.

Component What it does Implementation detail
Token embedding Maps token IDs to vectors. Lookup table shaped vocab_size x n_embd.
Position embedding Gives order information to otherwise permutation-blind attention. GPT-2 learns absolute position embeddings up to context length.
Causal self-attention Lets each position read earlier positions. Q, K, V projections, scaled dot products, triangular mask, softmax, weighted value sum.
Multi-head attention Runs several attention channels in parallel. Heads are concatenated then projected back to the residual width.
MLP Applies per-token computation after communication. Usually expands to 4x width, applies GELU, then projects back.
LM head Converts hidden states to logits over the vocabulary. Often tied to token embedding weights in GPT-style models.
attention(Q, K, V) = softmax((Q K^T) / sqrt(head_dim) + causal_mask) V
Module 05

The training loop is mostly engineering discipline

Once the model and loss exist, most work is making optimization correct, stable, and fast. Karpathy's GPT-2 reproduction steadily adds the production details missing from the toy notebook.

Optimization mechanics

  • Use AdamW with decoupled weight decay.
  • Decay only matrix weights, not biases and layer norm parameters.
  • Warm up the learning rate, then decay it, commonly with cosine decay.
  • Clip gradients when needed to avoid unstable spikes.
  • Accumulate gradients to simulate a larger token batch than fits in memory.

Systems mechanics

  • Use bf16 or fp16 compute while keeping numerically sensitive parts stable.
  • Enable TensorFloat-32 or fast matmul modes where appropriate.
  • Use fused kernels, FlashAttention, and torch.compile when they are correct.
  • Use Distributed Data Parallel across GPUs, one process per GPU.
  • Log train loss, validation loss, samples, eval scores, and checkpoint state.
# One training step
with autocast(dtype=bfloat16):
    logits = model(x)
    loss = cross_entropy(logits, y) / grad_accum_steps

loss.backward()
if micro_step_is_last:
    clip_grad_norm_(model.parameters(), max_norm)
    optimizer.step()
    scheduler.step()
    optimizer.zero_grad(set_to_none=True)
Module 06

GPT-2 reproduction gives concrete dimensions

The GPT-2 124M lecture turns the abstract GPT recipe into a specific target: load the released GPT-2 weights, implement an equivalent PyTorch module, verify that weight names and shapes match, then initialize from scratch and train.

Item GPT-2 124M setting Why it matters
Layers 12 Transformer blocks Depth controls repeated communication and computation.
Embedding width 768 channels Width controls representation size and matrix dimensions.
Heads 12 attention heads Each head is 64-dimensional when n_embd is 768.
Vocabulary 50,257 GPT-2 BPE tokens Determines embedding rows and final softmax classes.
Context 1,024 tokens Maximum prefix length seen by the model during pretraining.
Objective Next-token cross entropy The same basic objective used in the small Shakespeare GPT.

Evaluation caveat

Matching or beating GPT-2 on a modern run is not a pure architecture contest. Dataset distribution, data quality, leakage risk, tokenizer details, training schedule, and benchmark age all matter. Treat HellaSwag or validation curves as useful checks, not as final proof of general capability.

Module 07

A base model is a document completer, not an assistant

Pretraining makes a model that continues text from its training distribution. Chat behavior comes from later data and objectives. The model must learn a conversation protocol, when to answer, what tone to use, and when to refuse.

Supervised fine-tuning

Continue training on user/assistant transcripts. Often the loss is focused on assistant tokens so the model learns the response role.

Preference tuning

Collect rankings or comparisons, train a reward/preference model, and fine-tune outputs toward preferred answers.

Reasoning RL

For verifiable domains such as math and code, reinforcement learning can improve search-like thinking strategies beyond imitation.

Mental model

A normal chat model can be thought of as simulating the kind of answer a trained data labeler would write under the lab's instructions. A reasoning model adds an extra learned problem-solving process, especially when answers can be checked.

Module 08

Capability is bounded by compute, context, and evaluation

The lectures repeatedly separate what the model can do from what it reliably does. Larger models trained on more and better tokens usually improve, but failures remain uneven: hallucination, brittle arithmetic, forgotten context, special-token weirdness, and overconfident answers.

Where scaling helps

More parameters, more tokens, better data, better optimizers, and faster kernels can reduce loss and unlock downstream behavior. The GPT-2 reproduction shows the same code path can scale from a toy character model to a real 124M checkpoint.

Where scaling is not enough

Context is finite, models do not update their weights at test time, tokenizers create awkward blind spots, and benchmarks can be contaminated or gamed. Use models as tools with verification, not as sources of guaranteed truth.

Practice Path

Labs that match the lectures

These exercises are sequenced so each one produces a tangible artifact before moving to the next layer of complexity.

1. Train a character GPT

Use Tiny Shakespeare, a 65-character vocabulary, B x T batches, and a small decoder-only Transformer. Confirm loss drops and generation becomes Shakespeare-like.

2. Build a BPE tokenizer

Train byte-level merges on a small corpus, implement encode/decode, then compare token counts for English, code, numbers, and non-English text.

3. Match GPT-2 shapes

Implement a GPT-2 compatible module and inspect the state dict: token embeddings, position embeddings, attention projections, MLP weights, layer norms, and LM head.

4. Train a real token model

Use the GPT-2 tokenizer, pack documents with end-of-text separators, train on shards, and monitor validation loss and generated samples.

5. Scale the run

Add gradient accumulation, AdamW parameter grouping, learning-rate warmup and decay, mixed precision, FlashAttention, and DDP.

6. Post-train a chat variant

Convert conversations to a chat template, train on assistant completions, then evaluate whether the model follows the user/assistant protocol.
